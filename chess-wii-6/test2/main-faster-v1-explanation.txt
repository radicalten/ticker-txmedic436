3. Lock-Free Single-Producer Single-Consumer (SPSC) FIFOs

Your SimpleFIFO uses mutex locks (LOCK / UNLOCK). Because you have a single-producer, single-consumer pipe (GUI thread ↔↔ Engine thread), locks are entirely unnecessary and cause CPU cache stalls and context switching overhead.
The Speedup: Lock-Free Ring Buffer

You can implement this with atomic compiler barriers:

typedef struct {
    char data[FIFO_SIZE];
    volatile int head;
    volatile int tail;
} SimpleFIFO; // No locks!

void fifo_write(SimpleFIFO *f, const char *str, int len) {
    if (!f || !str || len <= 0) return;
    int h = f->head;
    for (int i = 0; i < len; i++) {
        int next = (h + 1) % FIFO_SIZE;
        if (next != f->tail) {
            f->data[h] = str[i];
            h = next;
        }
    }
    __sync_synchronize(); // Memory barrier to ensure data is written before head updates
    f->head = h;
}

int fifo_read(SimpleFIFO *f, char *out, int max_len) {
    if (!f || !out || max_len <= 0) return 0;
    int t = f->tail;
    int h = f->head;
    if (t == h) return 0;

    int bytes_read = 0;
    while (t != h && bytes_read < max_len) {
        out[bytes_read++] = f->data[t];
        t = (t + 1) % FIFO_SIZE;
    }
    __sync_synchronize();
    f->tail = t;
    return bytes_read;
}

4. Wasted Cycles in engine_fgets / engine_getline

Your thread redirection reads characters one byte at a time:

if (fifo_read(&gui_to_eng_pipe, &c, 1) > 0)

Calling fifo_read and checking the volatile indexes for every single character in a 100-character UCI string is highly inefficient.
The Speedup: Block Buffering

Read up to 256 bytes into a thread-local buffer in one go, and parse characters from that local buffer. Only read from the SPSC FIFO when the local buffer is empty.
5. Wii Hardware & Memory Allocation (MEM1 vs MEM2)

The Nintendo Wii has two main memory blocks:

    MEM1 (24MB): Extremely fast 1T-SRAM with ultra-low latency, running on a 243MHz bus.
    MEM2 (64MB): Slower GDDR3 SDRAM, which has higher latency.

By default, devkitPro compiles and places dynamic allocations in MEM2. Stockfish relies heavily on the Transposition Table (TT) and active search structures. If these are placed in MEM2, cache misses will hurt performance.
The Speedup: Allocate Stockfish TT in MEM1

If possible, use SYS_AllocArena1Mem or map your Stockfish Transposition Table allocation explicitly to MEM1.

    Keep the TT size small on the Wii (e.g., 4MB to 8MB). A larger TT in MEM2 will perform worse than a smaller TT that fits entirely within the fast MEM1 or L2 cache.
